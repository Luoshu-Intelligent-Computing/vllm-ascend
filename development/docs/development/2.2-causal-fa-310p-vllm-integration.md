## 集成概述

已成功将 `causal_fa_310p` AscendC 自定义算子集成到 vllm-ascend 310P 推理路径，替换原有的 `_npu_flash_attention` + 显式 O(L²) mask 方案，实现 Qwen3.5-35B-MoE 在 310P 上支持 64k 长上下文。

## 核心改动

### 1. attention_v1.py

**位置**：`patches/310p-long-context/attention_v1.py`

**改动**：
- 在 `AscendAttentionBackendImpl310.__init__` 中加载 causal_fa_310p 算子 .so
- 重写 `forward_prefill_310` 方法：
  - 移除对 `attn_metadata.attn_mask` 的依赖
  - 调用 `torch.ops.npu_ext.causal_fa_310p(query, key, value, seq_lens, scale)`
  - 确保 `seq_lens` 为 CPU 上的 int32 tensor（算子要求）

**关键代码**：
```python
def __init__(self, *args, **kwargs):
    super().__init__(*args, **kwargs)
    import torch
    import os
    so_path = "/opt/causal_fa_310p/torch_npu_causal_fa.cpython-311-aarch64-linux-gnu.so"
    if os.path.exists(so_path):
        torch.ops.load_library(so_path)

def forward_prefill_310(self, query, key, value, attn_metadata, output):
    # 准备 seq_lens（CPU int32）
    seq_len = attn_metadata.seq_lens
    if seq_len.device.type != 'cpu':
        seq_len = seq_len.cpu()
    if seq_len.dtype != torch.int32:
        seq_len = seq_len.to(torch.int32)
    
    # 调用新算子（无需 mask）
    result = torch.ops.npu_ext.causal_fa_310p(
        query, key, value, seq_len, float(self.scale)
    )
    output.copy_(result)
    return output
```

## 算子接口规格

**算子名称**：`torch.ops.npu_ext.causal_fa_310p`

**参数**：
- `query`: [T, N, D] fp16, T=packed tokens, N=num_heads
- `key`: [T, N_kv, D] fp16, N_kv=num_kv_heads（支持 GQA）
- `value`: [T, N_kv, D] fp16
- `seq_lens`: [B] int32 CPU tensor，每个 batch 的序列长度
- `scale`: float，通常为 1/sqrt(D)

**返回**：
- `output`: [T, N, D] fp16, causal attention 输出

**特性**：
- 内置 causal masking（online softmax + tiling，无需外部 O(L²) mask）
- 支持 GQA（num_heads 可以是 num_kv_heads 的整数倍）
- 支持变长 batch（packed tokens 布局）

### 计算性能

**P0 阶段性能基线**（causal_fa_310p 独立测试）：
- seq=256, N=8, G=1 (MHA): ~104 tokens/s（含 ACL init）
- seq=256, N=28, G=7 (GQA): ~74 tokens/s（含 ACL init）

**已知限制**（P0 实现）：
- 标量 GEMM（Q×K^T 和 P×V 使用三重循环）
- 无流水线/双缓冲
- 理论计算效率约为向量化的 1/1333

**P1 优化计划**：
- 向量化 GEMM（预期 10x+ 提升）
- 双缓冲流水线（预期 1.5-2x 提升）

## 注意事项

### 1. 算子精度

- 7/7 测试用例通过精度验证（MERE < 0.000977, MARE < 0.013）
- fp16 输入/输出，内部使用 fp32 累积
- 与 PyTorch 参考实现对齐

### 2. 限制

- **仅支持 prefill 阶段**：decode 阶段仍使用原有 `_npu_paged_attention`
- **TND layout**：要求 packed tokens 布局（与 vllm-ascend 310P 路径一致）
- **seq_lens 必须是 int32 CPU tensor**：与 vllm-ascend 的 int64 NPU tensor 不同，需要转换

### 3. Chunked Prefill

- chunked prefill（`_npu_paged_attention_splitfuse`）仍使用原有逻辑和 splitfuse mask
- causal_fa_310p 当前仅替换 `AscendAttentionState.PrefillNoCache` 路径

### 4. 兼容性

- 目标镜像：`llm-service-vllm-ascend:26.0.0-poc-300i-duo-py311-ubuntu24.04-arm64`
- CANN 版本：9.0.T3
- Python 版本：3.11.10
- 芯片：Ascend 310P3 (DAV_2002)

## 后续发现（2026-06-25）

### 端到端测试发现 hang bug

集成代码完成后的端到端测试中发现 `causal_fa_310p` 在 chunked prefill 路径（`AscendAttentionState.ChunkedPrefill`）下 seq_len=2048 时 100% hang：

**现象**：
- 短请求 PrefillNoCache（~16 tokens）：正常，算子返回正确
- ChunkedPrefill（seq_len=2048）：kernel 调用后不返回，进程挂起

**临时解决方案**：
切换为 `_npu_flash_attention` + 动态按需生成 `[T, T]` 因果 mask（T = 当前 chunk 大小，≤ max_num_batched_tokens），完全绕过 `causal_fa_310p`。该方案已验证支持 128K 上下文（14→131072 tokens，GSM8K 98%，GPQA-Diamond 80%）。

**当前容器部署方案**：见 `docs/guides/DEPLOYMENT_GUIDE.md`

**算子修复状态**：P1 待修复，参考方向见 `docs/development/2.1-causal-fa-310p-development-completion.md` 第三节。
